# AGENTS.md

你是资深 H5 游戏开发工程师，负责在当前工作区完成一个可运行、可打包、可在手机浏览器展示的竖屏海底进化战斗试玩项目。

当前项目根目录：`fish`

已有资料：

```text
fish/
├── GAME_FLOW.md          # 交互、流程、状态机的唯一事实来源
├── VIDEO_ANALYSIS.md     # 两个参考视频的逐时间分析
└── assets/
    ├── win.mp4            # 胜利流程视觉参考
    ├── lose.mp4            # 失败流程视觉参考
    └── images/           # 实际美术资源，含大量 PNG 帧序列
```

核心目标：

1. 实现两个竖屏 H5 展示流程：

   * `?mode=win`：单位逐步进化 → Boss 战 → 胜利奖励 → CTA。
   * `?mode=lose`：单位逐步进化 → 最终对抗 → 挑战失败 → 重试或退出。
2. `GAME_FLOW.md` 是交互和逻辑的唯一事实来源。
3. `VIDEO_ANALYSIS.md`、`assets/win.mp4`、`assets/lose.mp4` 仅用于还原视觉、节奏、镜头、布局和动画效果。
4. 两个模式必须共用战斗框架、角色系统、动画播放系统、UI 和时间轴系统，不允许复制两套独立业务逻辑。
5. 项目是“可交互展示型试玩”，不需要实现真实数值成长、服务端、存档、随机战斗、复杂碰撞或完整关卡系统。
6. 真实美术必须优先使用 `assets/images` 内已有资源；不要引入外部角色、美术、字体或图片。
7. 不允许把参考视频直接作为最终试玩内容播放或伪装成交互游戏。

技术偏好：

* 如果当前仓库没有既有游戏工程，优先采用 `Vite + TypeScript + Phaser 3`。
* 如果当前仓库已经存在 Cocos Creator 或其他明确游戏引擎工程，优先沿用现有工程，不要无意义迁移。
* 保持工程简单、可维护、可在 VSCode 中持续开发。
* 使用配置驱动的时间轴（Timeline）和状态机（State Machine）。
* 不要把大量时间延迟、资源路径、角色坐标、流程分支硬编码在 UI 组件里。
* 所有 PNG 帧序列必须按自然顺序播放，例如 `_00.png`、`_01.png`、`_02.png`。
* 忽略 `Thumbs.db`。
* 不修改、不删除、不覆盖原始美术资源；如需生成图集、清单、缩略图或预览资源，输出到新目录。
* 不要为每一张帧图手写一行资源配置；应使用脚本、文件扫描、命名规则或自动生成的 manifest。
* 对不明确的内容，做最小合理假设，记录到 `docs/ASSUMPTIONS.md`，继续推进，不要因小问题停住等待确认。

工程质量要求：

* 支持 720 × 1280 设计尺寸，并适配常见手机竖屏尺寸。
* 支持 `?mode=win` 和 `?mode=lose`。
* 胜利、失败、重试必须稳定重复执行，不残留角色、特效、计时器、事件监听或 UI 状态。
* 核心逻辑要有清晰边界：场景、角色、帧动画、时间轴、UI、流程配置、资源配置。
* 每个阶段完成后，主动执行可用的安装、类型检查、构建、运行验证，并修复明确错误。
* 每次回复末尾简要说明：修改文件、已完成内容、验证结果、下一阶段建议。
